RESUMEN COMPUTO 1

BASES DE DATOS NOSQL

¿Qué es NoSQL?

NoSQL es un enfoque para el diseño de bases de datos que puede adaptarse a una amplia variedad de modelos de datos, incluidos los formatos de clave-valor, documentos, columnas y gráficos. NoSQL, que significa "no solo SQL ", es una alternativa a las bases de datos relacionales tradicionales en las que los datos se colocan en tablas y el esquema de datos se diseña cuidadosamente antes de crear la base de datos.

Hoy en día, el movimiento NoSQL incluye cientos de productos de bases de datos NoSQL, lo que ha llevado a una variedad de definiciones para el término, algunas con muy principios comunes y otros no tan comunes. Bases de datos NoSQL también fueron creados para estructuras de datos y modelos que en un RDBMS requerían considerable gestión o trituración y la reconstitución de datos en código complejo.

Características

  • Un esquema de base de datos es la descripción de todos los posibles datos y estructuras de datos en una base de datos relacional. Con una base de datos NoSQL, no se requiere un esquema, lo que le brinda la libertad de almacenar información sin tener que diseñar un esquema por adelantado.
  • Las relaciones en una base de datos establecen conexiones entre tablas de datos. Por ejemplo, una lista de detalles de transacciones se puede conectar a una lista separada de detalles de entrega. Con una base de datos NoSQL, esto la información se almacena como un agregado: un solo registro con todo sobre la transacción, incluida la dirección de entrega
  • Algunas bases de datos están diseñadas para funcionar mejor (o únicamente) con hardware de procesamiento y almacenamiento especializado. Con un NoSQL base de datos, se pueden utilizar servidores baratos disponibles en el mercado. Añadiendo más de estos Los servidores baratos permiten escalar las bases de datos NoSQL para manejar más datos.

¿Cuándo se debe utilizar NoSQL?

  • 1. Cuando es necesario almacenar y recuperar una gran cantidad de datos.
  • 2. La relación entre los datos que almacena no es tan importante.
  • 3. Los datos cambian con el tiempo y no están estructurados.
  • 4. No se requiere compatibilidad con restricciones y uniones a nivel de base de datos.
  • 5. Los datos crecen continuamente y necesita escalar la base de datos con regularidad para manejar los datos.
